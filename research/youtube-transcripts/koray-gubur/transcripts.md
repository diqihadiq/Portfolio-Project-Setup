**Transcript:**

[On topical authority evolution - from controversy to mainstream]

When I launched my first case studies, even people claimed that I was using PBNs because they didn't believe that without links a website could rank just with semantic text. I don't blame them. But we didn't use PBNs or anything for ranking these websites.

I remember link building agencies were lynching me because they couldn't sell more links because semantics were becoming trendy. To make peace with them, whenever I go onto stage I explicitly say as the first thing that I also buy links. I'm happy to buy. I just say that if you implement semantics properly you might need fewer links - that's the main motto.

Back then basically it was a little bit more controversial and there were two different camps - link builders and the semantic lovers - and they were just fighting with each other. I was in the middle saying don't fight, both of them work, just do holistic SEO.

After especially Google's API leak - since there was directly a topical authority module inside Google's API leak, and they have seen page embedding, site embedding, site focus score, and topicality score type of things there too - and then Google's OG leak files came and there too Pandu explained basically topical authority together with PageRank and the click data. After that people accepted it. Today it's not a discussion anymore whether it is real or not.

[On Fortune 500 clients and common misconceptions]

These Fortune 500 companies, people usually assume they have the best of the best managers. Not at all. It's the opposite. If you work with a startup or medium-sized ecommerce, their managers and employees in my opinion and experience are way much better than Fortune 500 companies.

They care still about Semrush and Ahrefs type of metrics like DR or DA. They sometimes say for instance we increased our DR more than them. Why? It doesn't show the ranking difference because it's just a third party tool with a made-up metric.

I believe the most important thing in these big companies is training. You have to train them for around two to three months with these patterns, with these ranking algorithms, by explaining the processes. Then they actually start to function in a really good routine way.

The corporate work will be the hardest thing there for you. You will need to have really long meetings for really small things. I remember one instance that after 196 days a developer mailed me and said that a task is done. And I even forgot the task. More than six months we waited for that - two core updates passed.

[On AI content and strategic deployment vs reckless automation]

For this type of AI content publication, I suggest people to stay in the humanly range. Do not try to publish like 40,000 documents in one go. Try to stay actually in your human range so Google can understand that you are covering that topic by putting human effort in that.

In the API leak of Google, you will directly see there is a section that they are talking about content effort and it is a signal for ranking. If you go to the Google creator guidelines, they talk about total human effort either for designing or for writing.

With the last core update, old URLs are ranking better. Some URLs from Facebook from 2019 and 2022 are now ranking higher because I believe Google by using Reddit content they are training their language model in a very sensitive way to get real clean human dialogues.

The GPTs that I created were not for generating content. They are mainly for analyzing the content, analyzing the site, and accelerating the process of creation of the content. One of them is just about generating or extracting facts based on importance. One is about finding the most contextual predicates to use in the content. One is about extracting triples and assigning prominence score.

For AI content publication momentum - yes, it helps you rank higher but Google hits the website later. That's why it doesn't matter if you publish too many things. If a website has been already established with good amount of page rank, historical click data and good amount of coverage, it means that website is already safe and over a threshold. But if you publish a new project, you should be way much more careful because you are coming to that threshold from zero.

[On topical map creation - what remains exclusively human]

When you want to create a really good topical map, it's not that much automatable. You can create maybe a raw topical map - raw and processed topical maps are two different things. A raw topical map just gives you entities and the attributes. But how you will be shaping your micro context or specifically your H1s and the context of the page - it should most of the time be coming from you.

I usually try to explain this with a concept that I call topical map matrix. Let's say these four fingers represent four different websites and this section represents the query network that we want to rank for. Let's say the query network is about PHP developer. One website is about a web developer forum. One is about a university which has PHP development as a branch. One is a HR website for PHP developer salaries or job finding. And one is about overall PHP development and PHP software list or directory.

All four can rank for PHP developer - but all four have four different source types, source contexts, and four different internal links. The forum website is going to be linking to another community internally. University website can link to other branches. HR website links to other job findings.

If query processing is different - if Google processes PHP developer as PHP developer jobs - job website gets prominence. But in this case, the forum website will need to be shaping its topical map to compete because the query has a tendency to be closer to the job website.

In this matrix - what type of sources are ranking with what angle, how Google processes the query, what type of query network and templates are existing - from there, how you create this topical map will need to be changing. Usually when we create it in the perfect way, we create a topical map that covers all of these possible angles so that whatever Google thinks about this query, you can always stay rankable.

Creating this topical map in this way is not something that people can automate easily. AI still doesn't have that angle because there is not that much data that they can be trained on for these purposes. You should be processing the query as Google does. From there it gets easier.

[On AI search engines and dual domain strategy]

When I explain it, I said to people directly this: if you want to rank on the eyes of Google, you have to be cheap with higher quality. It's not about being more quality. It is about being cheaper for them to understand your content. The same thing happens for AI even heavier.

When you search something on Google, you press enter and Google's job ended. In AI it's not like that. You keep typing, you keep asking, you keep talking and creating computational resources from them a lot and a lot. That's why cost becomes more important. That's the reason I believe they are trying to use their own internalized index without checking the web - because if they use their own internal index, it gets cheaper for them.

For targeting AI answers, we sometimes open two different domains. One of them becomes our main money site which decreases the cost of retrieval a lot. The second domain is a kind of exact match domain and there we just publish content for semantic closeness. If we are able to match a certain paragraph to certain questions in ChatGPT, that passage can rank for that question. We don't care about whether it gets traffic from Google. If it is indexed by Google that's good enough. The rest is just for ChatGPT.

Another methodology: if I have a webpage, I can use no-index in the HTML for a section and tell Google not to index that section, and use it only for LLMs. Or I can have two different subfolders - one for regular systems, one that I disallow to Googlebot but keep open to all LLMs.

We also call these secondary domains "wasteful domains" - we add lots of content in them and we just expect LLMs to be fed with this data. And we use heavily social media platforms as well because when we use social media content it affects LLMs faster. For instance, I add podcast transcripts to IMDb. ChatGPT trusts IMDb a lot. So when I add my podcast and transcript to IMDb, I'm able to affect the answers in a really good way.

[On the concept of cannibalization - the best SEO advice that is actually wrong]

A few well-known people are saying cannibalization doesn't matter anymore. They say this because they say if you open too many pages, Google will understand many intent differences and always rank the right page even if the query phrases are too close to each other. I believe this is wrong. Cannibalization is a reality and it's not going to be working that way.

Just for ranking AI - do not open millions of pages. That's why we separated domains - wasteful domain is for the waste. When it comes to your real domain, manage it in a proper way. Cannibalization still a real thing. Don't think that it is okay to cannibalize things by trusting the capacity of AI.

Don't trust Google that they are smart. It's not a smart algorithm that much as you expect.

[On content effort and what Google actually measures]

In the API leak of Google, you will directly see there is directly a section that they are talking about content effort and it is a signal for ranking. Don't trust Google that they are smart.

I personally call it a glitch because in Southeast Asia or also in Brazil or Colombia, for unregulated industries in iGaming, in every 2-3 months Google basically stops serving the results for branded terms. The algorithm there is broken.

[On AI tools in SEO workflow]

Since technical artificial intelligence is a kind of machine interface - especially for teaching semantics in my opinion or seeing the anatomy of a document in the way that a machine can see it - it is a great tool. When I look at a Wikipedia page as a human, based on experience I can judge some stuff. But if I create a GPT and tell it to take all the predicates and take all the triples here, it's able to give me another layer that my eyes can't see directly.

I can also run five different relevance algorithms through the AI and get five different relevancy scores for a specific landing page. In terms of testing, observing, examining, analyzing - AI is doing a great job there.

But when it comes to giving decisions and especially communicating with internal teams, please don't use AI for that. Decisions shouldn't be given by AI at all. GPT is a great personalization of people who like to tell their boss what they want to hear. Whatever you say, GPT keeps saying it's a great question, great answer, you are right. It never says you are wrong. If something is always agreeing with you, it's not easy to trust the judgment it takes.

## Video 2

**Title:** AI & The Evolution of SEO in 2026 (Chat with Sean)

**Date:** March 2026
**Duration:** ~60 min

**Transcript:**

[On predictive information retrieval and cost of retrieval]

When I solved predictive information retrieval for the first time, I believe it was from Triston Upstill, one of the important Google engineers. He was saying that even if you wouldn't be evaluating the entire web document - just if you evaluate only the first four sentences or the first two sections - quality judgment doesn't change that much. Which means even if they evaluate the entire web document or just a small part of that, the ranking scores in their system doesn't change click satisfaction much.

He was trying to say this: cost of ranking a site can't be higher than cost of not ranking a site. So in other words, they try to focus on the cost before focusing on the quality. Even just last week, Google announced that they decreased the 14 megabyte HTML document limit to only two megabytes - meaning they become more protective for their actual resources.

Predictive information retrieval is about predicting the quality and the relevance of the page by just looking at certain annotations. One of these annotations is definitely centerpiece annotation. In the Google API leak, we see that they have annotations for section boundaries, sentence boundaries, or images. Maybe they have 30 different annotations and they just check these and assign a score for initial ranking.

And Pandu Nayak from Google showed in DOJ testimony that they are not running every algorithm for every web page. If they see enough level of clicks or enough level of topicality signal, then they run things like RankBrain.

From SEO point of view, we are trying to understand which part of the page the search engine will be focusing on to predict quality. If you're able to optimize these sections with the right annotations in the right way, then we can communicate with the search engine without losing the chance of being evaluated or ranked earlier.

[On visual semantics and chunking]

There are two engineers who are really important in Google - Michael Bendisky and Martin Naylor. They filed a few patents recently and they are mainly about the design and the visual blocks of the page.

When you look at chunking - let's say these four fingers represent four different sentences. When we focus on the text, we chunk them 1, 2, 3, 4. But according to their patent on layout-aware document understanding - before they even process your text, the first thing they check is visual blocks or visual components. If they see that two sentences are in the same block and two are in another block, they chunk in a different way. How you structure the page visually directly affects the relevance of these sentences to each other and how they chunk documents.

There is one more concept called structured information card. When you go to an ecommerce site or flight ticket site, they know that these websites always have a card for their products, and they can't process this data like processing an actual paragraph because it's not a paragraph. There's not even a single sentence in a card. That's why they first get the labels, the blocks, and from there they verbalize it.

In my opinion, this visual design and visual semantics will be really important to help for accurate chunking and accurate communication in terms of relevance configuration with the search engine.

[On how topical authority formula works]

When I start to read the patents, everything was actually about coming to the end of semantics. And when I started to create a concept of topical map, and when I attached it to the topical authority, the main logic was actually giving Google one source for one topic that covers everything in the best and cheapest way, so that they can learn that topic from that specific source or website.

The one thing that changed our strategy was the concept of historical data. Because even if we were directly covering every topic right now, without Google spending enough level of time or doing click tests, they are not going to show you the results. And usually these results come after a core update.

If you know when the core update is going to happen, and if you are able to make your website ready before 20 days of that, you can give enough level of historical data to Google to replace the already existing ranking sources.

The formula: topical authority = topical coverage multiplied by historical data, divided by cost of retrieval. Which means if you increase the cost, it doesn't matter how much you cover for how long. But if you decrease the cost, increase the coverage and increase the amount of historical data, then you exceed the thresholds and search engine starts to test you and starts to approve the source.

There's just one more point. If you start to rank really high, just ensure that you are satisfying the clicks. If Google realizes that when people click onto your site the clicks are not satisfied, after a point based on their click satisfaction models they can see they should revert the rankings back.

This is called by them "link value proposition" - search engine result page link. Technically they say that if they link you from their result page, you should be proposing a value back.

[On query augmentation and AI fan-out]

Query augmentation is very related to AI as well because what people call query fan-out is actually query augmentation. When you search for any query, Google takes that query and verbalizes it - maybe 100 times in a different way - and they distribute possibilities. Because query language and natural language are different. Semantics is closing the gap between query language and natural language.

When you have a query that you use in the search language, Google needs to process and augment it in the way that you use in your daily life. From there they create all these AI answers or these indices or search engine result pages.

[On entity-attribute-value relationships]

There are two types of triples. One is subject-predicate-object. The other is entity-attribute-value. The value part is especially important because when you ask for somebody's birthday - the person is the entity, birthday is the attribute, but the actual date is the value. If you have usually the wrong values for your attributes, Google can't trust you. It is about consensus - if you are outside of the consensus, it's harder to rank you.

When it comes to the EAV system, you should always be in the consensus or you should be the source of the consensus, which is highly related to topical authority methodology.

[On social media and its influence on SEO rankings]

At 8th December 2025, Google announced that they are integrating social media channels directly into Search Console. We already have some projects that have their YouTube channel in Search Console and it is giving us number of clicks that videos are taking from YouTube. During that time, it increased clicks by 60% - YouTube video clicks from Google increased 60%.

In my opinion, since 2019, if you have an official YouTube channel that is active and if you are really active in socials, it directly helps your rankings. Whenever we get a viral video from TikTok for hair transplant, our clinics rank higher for all of the terms because popularity is one of the main metrics.

You are not ranking your website. You are ranking your web entity. Your website, your TikTok, your Facebook, your YouTube - they are actually all part of your web entity. If your web entity has more activity, it's going to be a huge help.

We are automating Reddit in a very aggressive way. Whenever there is a new submission for a listing in real estate, we post it onto Reddit and link from Reddit back. We see that it's increasing crawl stats and indexation speed through social links and social media shares.

[On AI tools and their limitations]

I believe AI should be used as a kind of local database that you control. You shouldn't expect it to replace a human brain. They can do lots of things very fast, but when it comes to creativity or understanding or having the feeling of responsibility, that's something different. AI won't feel guilty when it says something wrong to you. It just wants to save that specific computation and that iteration.

There was even a research saying that people who use AI a lot, their brain activity levels were actually lowering. It might be bad in the long term. It will make people less responsible, lazier, and probably less patient especially. I believe there should be some regulations that governments should be performing especially for young kids or people who are in the development age.

## Key Themes from These Transcripts

- Topical authority formula: Topical Coverage x Historical Data / Cost of Retrieval - all three must be optimized together
- Cost of retrieval is potentially more important than topical authority itself - Google must find it cheaper to rank you than to not rank you
- Google recently decreased HTML document crawl limit from 14MB to 2MB - they are increasingly protecting computational resources
- Visual layout directly affects how Google chunks content - layout-aware document understanding patent is critical
- Topical map matrix: four different source types can rank for the same query with different contexts - create a topical map that covers all possible query processing angles
- AI content deployment strategy: stay in "humanly range" of publication, use second "wasteful domains" for LLM feeding without risking main domain
- ChatGPT, IMDb, Reddit, LinkedIn - all viable sources to influence LLM citation answers
- Cannibalization still matters - experts saying otherwise are wrong according to Koray
- Don't trust Google's algorithm to be smart - it processes content based on embeddings and can make absurd ranking decisions
- Historical data + core update timing is the key to ranking - prepare sites 20 days before expected core updates
- Query fan-out = query augmentation - Google verbalizes queries 100+ different ways before matching to documents
- Social media is now officially part of SEO - web entity includes all social profiles, activity on any platform affects rankings
- Entity-attribute-value consensus: wrong values for attributes damage rankings - you must align with or become the source of consensus
- AI is excellent for analysis, extraction, and pattern recognition but should NEVER be the decision maker
